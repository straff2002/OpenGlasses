## Design Context

### Users
OpenGlasses serves a broad range: tech enthusiasts experimenting with smart glasses, everyday users wanting a seamless hands-free assistant, and developers extending the platform. The common thread is that users are wearing Ray-Ban Meta glasses and interacting primarily by voice — the phone screen is secondary, glanced at briefly. The interface should feel like a calm dashboard that confirms the system is working without demanding attention.

### Brand Personality
**Calm, capable, invisible.** The UI should recede — the assistant just works. When you glance at your phone, you see status at a glance and nothing more. The design should inspire confidence that everything is running smoothly without requiring active management.

### Aesthetic Direction
- **Visual tone**: Dark, ambient, minimal. The app is a black canvas with subtle glowing status indicators — like an instrument panel in a dark room.
- **Reference**: Apple Watch / Siri — status-driven ambient UI that fades away when not needed. Information surfaces only when relevant.
- **Anti-reference**: Busy dashboards, chatbot UIs with message bubbles, skeuomorphic controls, anything that demands sustained visual attention.
- **Theme**: Dark mode only. The existing full-black background with glass-morphism controls and cyan/orange accent palette is the foundation.
- **Color system**: Cyan (active/listening), orange (speaking/connecting/warning), red (error/recording), green (streaming/success), gray (idle/disconnected), purple (tool execution). Muted variants for secondary UI in settings.
- **Typography**: System font (SF Pro), monospaced for technical readouts. Light weight for large ambient elements, medium for labels, regular for body.
- **Iconography**: SF Symbols exclusively. Light-to-medium weight, matching the ambient feel.

### Design Principles
1. **Glanceable over readable** — Users look at the phone for 1-2 seconds. Every screen state should communicate its meaning in that window. Prefer large icons and color-coded status over text.
2. **Ambient, not interactive** — The main screen is a status display, not a control panel. Controls exist but stay quiet. The voice is the primary interface; the screen is secondary confirmation.
3. **Dark canvas, living light** — The black background is the constant. Meaning comes from colored light: pulsing rings, glowing icons, subtle material overlays. No flat colored backgrounds.
4. **Progressive disclosure** — Settings and configuration are deep but never in the way. The main view is radically simple; complexity lives behind deliberate navigation.
5. **Trust through transparency** — Show connection state, active mode, tool execution, and errors clearly. The user should never wonder "is it working?" — ambient indicators answer that question continuously.
6. **Platform-native UI** — Use standard iOS components (Form, NavigationStack, Section, Toggle, Picker, etc.) wherever possible. Settings, sheets, and configuration screens should feel like a first-party iOS app, not a custom design system. Reserve custom UI only for the ambient main view where standard components don't fit.

### Accessibility
Standard iOS defaults (Dynamic Type, VoiceOver, reduce motion). Rely on system accessibility features. Ensure sufficient contrast for status colors against the dark background.
